# 电商平台优化第一期 3

---

## 一、优化概述

本文档整合《电商平台第一期优化.md》与《电商平台第一期优化2.md》的内容，针对 `admin-backend` 和 `admin-web` 电商模块进行系统性分析，从**业务功能、代码组织、性能、安全性、可维护性**五个维度提出优化建议，形成可执行的第一期优化路线图。

**文档定位**：
- 继承第一期优化文档的技术方案（状态枚举、状态机、库存反向操作等）
- 融合第二版文档的审计结论、已落地能力、优先级划分、Quick Wins

## 二、模块现状概览

### 2.1 已形成闭环的能力

```text
商品(SPU/SKU) → 上架链接 → 销售订单 → 库存扣减/欠货
                    ↓
              Excel 导入(sys_import_profile)
                    ↓
         月结统计 ← 快递账单导入 → 运单号回填真实运费
                    ↓
              快递站点(价格/须知/别名/面单价)
```

### 2.2 核心模块清单

| 领域 | 关键类/页面 | 说明 |
|------|-------------|------|
| 销售订单 | `EcSalesOrderServiceImpl`、`SalesOrderPanel.vue` | CRUD、导入、发货、退款 |
| 导入配置 | `SysImportServiceImpl`、`ImportMappingDialog.vue` | `SALES_ORDER` / `SETTLEMENT_EXPRESS_BILL` / `EXPRESS_STATION_NAME` |
| 月结 | `EcMonthlySettlementServiceImpl`、`MonthlySettlementPanel.vue` | 汇总、买家排除、账单导入 |

| 库存 | `EcInventoryServiceImpl`、`InventoryPanel.vue` | 库存、进货、盘点、流水 |

### 2.3 本阶段已落地优化

#### 2.3.1 快递站点导入名称别名
- `sys_import_profile`：`biz_type = EXPRESS_STATION_NAME`，`scope_key = express_station:{站点ID}`
- 解析顺序：精确匹配 `ec_express_station.name` → 别名映射

#### 2.3.2 订单重复导入覆盖（非追加）
- 同店铺 + 同 `platform_order_no` 已存在 → 覆盖整单
- 同单内按明细键去重

#### 2.3.3 1688 卖家备注 → 手动填写成本
- 仅 1688 平台：任一明细行有卖家备注 → 整单所有行 `matchStatus = UNMATCHED`

#### 2.3.4 快递账单导入
- 明细表、列映射统一到 `sys_import_profile`
- 重复导入同月同站点同运单号 → 覆盖旧值

### 2.4 主要短板（审计结论）

| 类别 | 描述 |
|------|------|
| 数据一致性 | 导入「已发货/已完成」未扣库存；导入退款未算亏损 |
| 并发安全 | 库存扣减 read-modify-write，无原子 SQL / 乐观锁 |
| 删除约束 | `deleteOrder` 未限制状态，未级联清理欠货/扣减 |
| 前端闭环 | 订单 confirm/ship/refund API 已有，UI 未接入 |
| 性能 | 8 Tab 同时挂载首屏多请求；导入 N+1 链接匹配 |
| 可维护性 | `EcSalesOrderServiceImpl` ~1700 行；状态字符串三套并存 |

### 3.3 库存反向操作（高优先级）

**问题**：退款/取消订单时没有将库存加回，欠货记录没有核销接口。

**优化方案**：

```java
// EcSalesOrderInventorySupport 新增方法
public void reclaimSkuCode(Long orderId, Long orderLineId, String skuCode, int reclaimQty) {
    EcInventory inventory = ecInventoryMapper.selectOne(...);
    if (inventory != null) {
        inventory.setQuantity(inventory.getQuantity() + reclaimQty);
        ecInventoryMapper.updateById(inventory);
        
        EcInventoryLog log = new EcInventoryLog();
        log.setChangeType("RECLAIM");
        log.setChangeQty(reclaimQty);
        ecInventoryLogMapper.insert(log);
    }
    clearShortage(orderId, orderLineId, reclaimQty);
}

public void clearShortage(Long orderId, Long orderLineId, int clearedQty) {
    EcSalesOrderShortage shortage = ecSalesOrderShortageMapper.selectOne(...);
    if (shortage != null) {
        shortage.setClearedQty(shortage.getClearedQty() + clearedQty);
        if (shortage.getClearedQty() >= shortage.getShortQty()) {
            shortage.setStatus("CLEARED");
        }
        ecSalesOrderShortageMapper.updateById(shortage);
    }
}
```

## 四、待优化项（按优先级）

### 4.1 P0 — 数据正确性

| 项 | 说明 | Quick Win |
|----|------|-----------|
| 导入 SHIPPED/COMPLETED 未扣库存 | `applyImportedLineStatus` 只改状态，不扣库存 | 导入 commit 后调用 `fulfillLine()` |
| 库存扣减无并发控制 | read-modify-write 模式无保护 | `UPDATE quantity = quantity - ? WHERE sku_code = ? AND quantity >= ?` |
| 订单删除缺少约束 | `deleteOrder` 未限制状态，未级联清理 | 限制「仅草稿可删」 |
| 导入 REFUNDED/RETURNED 未算亏损 | 导入路径只设 status，不算 `lossAmount` | 复用 `refundLine` 定价逻辑 |

### 4.2 P1 — 体验与性能

**后端**：

| 项 | 说明 | Quick Win |
|----|------|-----------|
| 导入 N+1 | 每行 `matchLinkSku` 多次 DB 查询 | 按 `shopId` 预加载 link/sku Map |
| 库存汇总 | 全量 list 再内存 SUM | 改 SQL `SUM` |
| 运单号匹配 | `REPLACE(tracking_number)` 无法走索引 | 入库时规范化，去掉 REPLACE |
| 状态常量重复 | Service 仍硬编码字符串 | 使用枚举 |

**前端**：

| 项 | 说明 | Quick Win |
|----|------|-----------|
| Tab 非懒加载 | 8 面板同时 `onMounted` | 加 `lazy` 属性 |
| 订单操作 UI 缺失 | confirm/ship/refund API 未接入 | 接入 API |
| 列表无空状态 | 仅 loading | 加 `el-empty` |
| i18n 缺口 | en-US 缺键 | 补全 en-US |

### 4.3 P2 — 架构与可维护性

**后端**：

| 项 | 说明 |
|----|------|
| God Class | `EcSalesOrderServiceImpl` 承担过多职责 |
| 状态机分散 | 导入/手工/API 三条履约路径逻辑不一致 |
| SQL 分散 | 多个 `ecommerce_*.sql`，新环境易漏执行 |
| 无外键 | 孤儿数据风险 |

**建议拆分**：
```text
EcSalesOrderServiceImpl
  → OrderCommandService      # CRUD、确认、取消
  → OrderImportService       # 预览、commit、覆盖
  → OrderFulfillmentService  # 发货、扣库存、欠货
  → OrderPricingFacade       # 行定价、重算、退款亏损
```

**前端**：

| 项 | 说明 |
|----|------|
| 大组件 | `SalesOrderPanel`(1072)、`ProductPanel`(874) |
| Options 重复拉取 | `fetchFactoryOptions` 多组件各自调用 |
| 表单校验 | 全模块无 `el-form :rules` |

### 7.3 表单草稿自动保存

```typescript
// composables/useDraftForm.ts
export function useDraftForm<T>(key: string, defaultForm: T) {
  const form = reactive<T>({ ...defaultForm });
  const STORAGE_KEY = `draft_${key}`;

  onMounted(() => {
    const saved = localStorage.getItem(STORAGE_KEY);
    if (saved) Object.assign(form, JSON.parse(saved));
  });

  watch(form, (newVal) => {
    localStorage.setItem(STORAGE_KEY, JSON.stringify(newVal));
  }, { deep: true });

  return { form, clearDraft: () => localStorage.removeItem(STORAGE_KEY) };
}
```
