# 笔记本功能规划

---

## 一、背景与定位

当前前端路由 `/notebook` 仍为占位页（`PlaceholderView`），产品描述为：**记录想法、待办与项目笔记，支持多端同步**。

笔记本在本项目中更适合定位为 **个人知识工作台**，而非 Notion 级协作平台：

- 服务日常场景：想法、待办、项目备忘、电商运营笔记
- 与后续 **AI 知识库**、**资料库** 模块打通，而非一开始就做成大而全的文档系统
- 实现风格对齐 **番茄钟**：单模块、Tab 分区、后端 CRUD + 前端 Element Plus

**相关代码位置：**

| 层级 | 路径 |
|------|------|
| 前端路由 | `admin-web/src/router/index.ts` → `/notebook` |
| 功能卡片 | `admin-web/src/data/function-items.ts` |
| 国际化 | `admin-web/src/i18n/locales/zh-CN.ts`、`en-US.ts` |
| 参考模块 | 番茄钟 `PomodoroPlanController`、`sql/pomodoro.sql` |

## 二、功能清单（按优先级）

### 2.1 第一期 MVP（建议 2～3 周落地）

| 功能 | 说明 |
|------|------|
| **笔记本 / 分组** | 多级笔记本（如：工作、电商、生活），支持排序、图标/颜色 |
| **笔记 CRUD** | 标题 + 正文，创建/编辑/删除/软删除 |
| **列表与搜索** | 左侧树或列表，右侧编辑区；按标题/正文关键词搜索 |
| **置顶 / 收藏** | 常用笔记快速访问 |
| **标签** | 多标签筛选（如 `#电商` `#待办` `#灵感`） |
| **最近编辑** | 按 `update_time` 排序的「最近」列表 |
| **自动保存** | 编辑防抖保存（如 1～2 秒），避免丢内容 |
| **回收站** | 软删除 + 恢复（与项目 `deleted` 逻辑一致） |

### 2.2 第二期（体验增强）

| 功能 | 说明 |
|------|------|
| **Markdown 编辑** | 正文支持 Markdown 预览（标题、列表、代码块、链接等） |
| **待办清单** | 笔记内 `- [ ]` 任务项，或独立「待办」类型笔记 |
| **笔记类型** | `NOTE`（普通）、`TODO`（待办）、`MEMO`（速记） |
| **关联业务** | 笔记可关联电商实体（店铺、链接等 ID），方便运营备忘 |
| **附件** | 复用现有图片上传能力，笔记内插图 |
| **导出** | 单篇 Markdown / 整本导出 |
| **版本历史** | 每次保存留快照，可回滚（轻量：只保留最近 N 版） |

### 2.3 第三期（与生态联动）

| 功能 | 说明 |
|------|------|
| **多端同步** | Web + 副屏只读展示「今日待办/置顶笔记」（ESP 可后续接 API） |
| **AI 辅助** | 摘要、润色、从笔记生成待办（为 AI 知识库铺路） |
| **全文检索** | MySQL `FULLTEXT` 或后续接 Elasticsearch |
| **模板** | 会议记录、日报、电商复盘等预设模板 |
| **分享链接** | 只读分享（若有多用户需求再上） |

## 三、页面结构建议

对齐番茄钟 Tab 式布局：

```text
笔记本
├── 全部笔记      # 时间线 / 列表
├── 我的笔记本    # 左侧分组树 + 右侧编辑
├── 待办          # 勾选式任务（可从第二期独立出来）
├── 标签          # 标签云 + 筛选
└── 回收站
```

**布局推荐：** 双栏结构 — 左侧约 280px 导航（笔记本树 + 标签），右侧编辑区（标题 + 富文本/Markdown + 元信息）。

## 四、数据模型设计

### 4.1 表结构（第一期）

```text
nb_notebook        # 笔记本分组
  id, name, icon, color, sort_order, user_id, deleted, create_time, update_time

nb_note            # 笔记
  id, notebook_id, title, content (TEXT), note_type, is_pinned, is_favorite,
  status (DRAFT/PUBLISHED), deleted, create_time, update_time

nb_note_tag        # 标签
  id, name, color, user_id, deleted, create_time, update_time

nb_note_tag_rel    # 笔记-标签多对多
  note_id, tag_id

nb_note_revision   # 可选，第二期做版本历史
  note_id, content, title, revision_no, create_time
```

### 4.2 与现有规范对齐

- 实体使用 `@TableLogic` 软删除
- 统一返回 `ApiResult` / `PageResult`
- SQL 脚本放置于 `admin-backend/sql/notebook.sql`
- Controller 自动走 `ControllerLogAspect` 接口日志

### 4.3 笔记类型枚举（建议）

```java
public enum NbNoteType {
    NOTE,   // 普通笔记
    TODO,   // 待办
    MEMO    // 速记
}
```

### 4.4 业务关联（第二期）

笔记可扩展关联字段，便于与电商模块联动：

```text
ref_type   # 如 EC_SHOP、EC_LISTING_LINK、EC_PRODUCT
ref_id     # 关联业务主键
```

## 五、技术选型建议

| 层级 | 建议 |
|------|------|
| **后端** | 与番茄钟同结构：`NotebookController`、`NoteController`、`TagController` |
| **正文存储** | 第一期纯 `TEXT`；大内容或版本历史再考虑拆分 |
| **前端编辑** | MVP 用 `el-input type="textarea"`；第二期接 `@vueup/vue-quill` 或 `md-editor-v3` |
| **状态管理** | 单页内 `ref` 即可，不必上 Pinia（除非跨页面共享草稿） |
| **搜索** | 第一期 `LIKE` + 索引；笔记量大再上全文检索 |
| **权限** | 当前单用户可先不加 `user_id` 过滤；有 `SysUser` 后预留字段 |

## 六、与现有模块联动

| 模块 | 联动方式 |
|------|----------|
| **电商** | 笔记 `ref_type` + `ref_id` 关联店铺/链接；在业务详情页展示「相关笔记」 |
| **番茄钟** | 专注结束后可一键生成「本次专注心得」笔记 |
| **AI 知识库** | 笔记标记「纳入知识库」，后续做向量化索引 |
| **ESP 副屏** | 只读 API 返回置顶待办若干条，不必第一期做编辑 |

## 七、开发分期

```text
第一期 MVP（笔记本 + 笔记 + 标签 + 搜索）
        ↓
第二期（Markdown + 待办 + 附件 + 版本历史）
        ↓
第三期（AI + 副屏 + 知识库联动）
```

### 7.1 第一期交付标准

- [ ] 执行 `sql/notebook.sql`
- [ ] 后端 3 组 API：笔记本、笔记、标签
- [ ] 前端 `NotebookView.vue` 替换 `PlaceholderView`
- [ ] 中英文 i18n
- [ ] 自动保存 + 回收站

### 7.2 建议刻意不做（避免范围膨胀）

- 实时协同编辑
- 复杂块编辑器（Notion 式）
- 第一期就上 AI
- 复杂文件夹嵌套（先 2 级：笔记本 → 笔记）

## 八、API 草案（第一期）

```text
GET    /api/notebooks                         # 笔记本列表
POST   /api/notebooks
PUT    /api/notebooks/{id}
DELETE /api/notebooks/{id}

GET    /api/notes?notebookId=&keyword=&tagId=&pinned=
GET    /api/notes/{id}
POST   /api/notes
PUT    /api/notes/{id}
DELETE /api/notes/{id}
POST   /api/notes/{id}/restore                # 回收站恢复

GET    /api/note-tags
POST   /api/note-tags
PUT    /api/notes/{id}/tags                   # 设置标签
```

### 8.1 请求/响应示例

**创建笔记：**

```json
POST /api/notes
{
  "notebookId": 1,
  "title": "电商复盘",
  "content": "本周链接调价记录...",
  "noteType": "NOTE",
  "tagIds": [1, 2]
}
```

**列表查询：**

```json
GET /api/notes?notebookId=1&keyword=调价&page=1&pageSize=20

{
  "code": 0,
  "data": {
    "list": [...],
    "total": 42,
    "page": 1,
    "pageSize": 20
  }
}
```

## 十一、设计原则总结

| 原则 | 说明 |
|------|------|
| **先能用再好用** | 第一期目标是「快速记、好找、不丢」，编辑器从简 |
| **对齐番茄钟** | 同一套后端分层、SQL 脚本、Tab 式前端，维护成本低 |
| **为 AI 知识库留接口** | 笔记结构保持纯文本/Markdown，便于后续检索与向量化 |
| **单用户先行** | 不做权限、分享，加快上线；字段预留 `user_id` |

## 十二、待确认事项

开发启动前建议确认：

1. **第一期范围**：更偏向「纯笔记」还是「笔记 + 待办一体」？
2. **编辑器**：第一期是否接受纯文本，还是必须 Markdown？
3. **多用户**：是否在 MVP 阶段就按 `user_id` 隔离数据？

确认后可按本文档第一期清单依次落地 `sql`、后端接口与前端页面。
